## 2. フラッシュテスト中にエラーが起きると無限ループ 【解消済み: 2026-05-29】

ステージコードは RAM (`0x2003E000`) で動作するが、`rtt_log_error` 内の文字列参照は
元のフラッシュアドレスを使う。

```asm
ldr r0, =rtt_msg_err   ; リテラルプールには flash 上の絶対アドレスが入る
bl rtt_write_str
```

現 ELF (`build/main.elf`, 2026-05-29 11:51 ビルド) のシンボルアドレス:

```
flash_stage_start  0x0000015c   ← ページ 0 内
rtt_id_string      0x00000524   ← ページ 0 内
hex_chars          0x00000534   ← ページ 0 内
rtt_msg_start      0x00000544   ← ページ 0 内
rtt_msg_err        0x00000557   ← ページ 0 内
rtt_msg_page_short 0x0000055c   ← ページ 0 内
flash_stage_end    0x000005b0   ← ページ 0 内 (0x000–0xFFF)
```

ステージコード全体がフラッシュ上のページ 0 に収まることが確認された。
フラッシュテストの最初の操作 `flash_erase_page(0x00000000)` でページ 0 が消去された後は、
`rtt_log_error` が呼ばれると `rtt_write_str` が `0xFF` を読み続けて永久ループに陥る。
その結果、フラッシュ操作のエラーがあっても `TEST_STATE_FAIL` がセットされず、
LED も変化せず、監視スクリプトはタイムアウトまで待つだけになる。

**補足**: J-Link スクリプトによる RAM ダンプとオンボード LED の目視を組み合わせれば
テスト結果の人手による読み取りは可能。ただし自動判定の信頼性は損なわれる。

2026-05-29 の更新で、`rtt_log_error` 内の文字列参照にリロケーションオフセット `r11`
を加算し、RAM 上のコピーを参照するよう修正した ([src/main.S:488-494](src/main.S#L488-L494))。
これにより、ページ 0 消去後でも失敗経路が `TEST_STATE_FAIL` に到達する。

2026-05-29 の再更新で、`rtt_write_hex32` ([src/main.S:462](src/main.S#L462)) も
`hex_chars` にリロケーションオフセット `r11` を加算するよう修正した。これでページ 0
消去後のエラーログでも 16 進値が正しく表示される。

同じフラッシュアドレス参照が残っている `rtt_msg_start` と `rtt_id_string` は、
いずれもステージ起動直後（ページ 0 消去前）にのみ呼ばれるため実害なし。

## 3. RAM テストの妥当性分析

### テスト構造

2 パスで構成される。

**パス 1** ([src/main.S:135-152](src/main.S#L135-L152)): 各ワードを低位から順に走査:

| パターン | 値 | 読み返し | 目的 |
|---------|-----|:------:|------|
| 1 | `0xAAAAAAAA` | 即時 | 全ビット 1→0 / 0→1 の確認 (偶数ビット=1) |
| 2 | `0x55555555` | 即時 | パターン 1 の補数 (奇数ビット=1) |
| 3 | アドレス値 | パス 2 で確認 | アドレスごとに異なる値を書き置く |

**パス 2** ([src/main.S:154-166](src/main.S#L154-L166)): 先頭に戻り全ワードを読み返して自アドレス値と照合。

### 検出できる故障 ✓

**スタック故障 (Stuck-At Fault)**  
パターン 1 (0xAAAAAAAA) と 2 (0x55555555) の組み合わせで 32 ビット全ビット位置の
`0→1` / `1→0` 遷移を要求する。あるビットが固定値に張り付いていれば必ず不一致が起きる。

**アドレスエイリアス故障** ([src/main.S:154-166](src/main.S#L154-L166))  
2026-05-29 に 2 パス構成を追加して対応した。

- パス 1: 全アドレスに 0xAAAAAAAA・0x55555555 を書いて即時検証、最後に自アドレス値を書く
- パス 2: 全アドレスを読み返し、各アドレスに自アドレス値が保持されているか確認

2 つのアドレス A・B がエイリアスするケースの動作:
1. パス 1 で A を書く → エイリアスセルに A が入る
2. パス 1 で後続の B を書く → エイリアスセルが B に上書きされる
3. パス 2 で A を読む → B が返る → A ≠ B → `ram_test_ng` ✓

**Cortex-M4 での即時読み返しは実際の SRAM から読んでいる**  
ARM Cortex-M4 TRM (DDI0439) Section 3.3.2 に、STR と LDR が同一アドレスの場合
LDR は STR の完了まで stall すると明記されている。ライトバッファからの forwarding ではなく
SRAM セルからの読み出しが保証されるため、DSB なしでも読み返しテストとして成立している。

### 検出できない故障 ✗

**カップリング故障**  
セル A への書き込みがセル B の値を変化させるケース。
A の直後に A しか読み返さないため、B への影響を見逃す。

### 実用上の評価

| 故障モード | 検出 | nRF52840 での発生可能性 |
|-----------|:----:|----------------------|
| スタック故障 (SA0/SA1) | ✓ | 電源不良・ESD・製造不良で発生する主要故障モード |
| 遷移故障 | ✓ | 上記に準ずる |
| アドレスエイリアス | ✓ | 合成 SRAM でのアドレスデコーダ故障は稀だが 0 ではない |
| カップリング故障 | ✗ | オンチップ SRAM では DRAM より発生頻度が低い |

nRF52840 のオンチップ SRAM は標準セルベースの合成 SRAM。
スタック故障・遷移故障・アドレスエイリアス故障を検出でき、「電源や実装に問題がないか確認する」
レベルの健全性チェックとして妥当。

## 4. テスト範囲の盲点 【設計上の制約】

RAM 上位 12 KB (`0x2003D000–0x20040000`) がテスト対象外。

| アドレス | 用途 | テスト |
|---------|------|:------:|
| `0x20000000–0x2003CFFF` | 汎用 RAM | ✓ |
| `0x2003D000–0x2003DFFF` | RAM ベクタテーブル (RAM_VECTOR_BASE) | ✗ |
| `0x2003E000–0x2003FFFF` | ステージコード実行領域 (RAM_STAGE_BASE) | ✗ |
| スタック領域 | `0x20040000` から下方向 | ✗ |

ステージの実行領域とスタックを守るための意図的な除外。README にも記載あり。
ただし「このテストが通っても上位 12 KB の欠陥は見逃す」という事実は明示しておくことを推奨する。

## 5. その他の軽微な問題

### `flash_page_log_skip` とページログ計算 【解消済み: 2026-05-29】

2026-05-29 の更新で、未使用のページログ計算と分岐残骸 (`lsrs / ands / b flash_page_log_skip`)
を削除した。ラベル名 `flash_page_log_skip` は残っているが ([src/main.S:269](src/main.S#L269))、
現在は ACL チェックコードの入り口を指すだけであり機能上の問題はない。

### `rtt_msg_pass` の RTT 出力 【解消済み: 2026-05-29】

2026-05-29 の再更新で、`stage_pass` から `rtt_msg_pass` を RAM コピー側参照で出力するようにした。
これにより PASS 時にも RTT 上で `PASS` を確認できる。

### ACL チェックが永続的に無効

`flash_page_acl_mode` は常に `0` を返す。
ACL でロックされているページを通常ページとして書き込もうとして NVMC が書き込みを
拒否した場合、verify が失敗してテストが止まる可能性がある。
README に「今後再有効化予定」と記載あり。
